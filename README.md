# 魔札錬成工房

2〜4人がスマートフォンのブラウザだけで遊べる、Gartic Phone形式のTCG風カード共作パーティーゲーム。

全員が1枚ずつオリジナルモンスターカードを担当し、全8ラウンドでカードを隣のプレイヤーへ回しながら「名前 → ステータス → テキスト → イラスト」を協力（すれ違い）制作。最後に完成カードを演出付きで発表します。

- サーバー: Node.js + Express + Socket.io（同一HTTPサーバー・同一ポート）
- フロント: Vite + React（SPA、スマホ縦画面最優先、PWA対応）
- 状態管理: サーバーが正。Redis対応（未設定時はメモリ管理へフォールバック）

設計の詳細は [DESIGN.md](./DESIGN.md) を参照してください。

---

## 必要環境

| ソフトウェア | バージョン |
|---|---|
| Node.js | **18.17以上**（推奨: 20 LTS / 22 LTS） |
| npm | 9以上（Node同梱のもので可） |
| Redis（任意） | 6以上 / Redis互換ストレージ（Upstash等）可 |

> **ルーム状態の保存について**
> - `REDIS_URL` 未設定時は**メモリ管理**になります。**サーバー再起動で進行中のゲームは消えます。**
> - 安定運用（本番）では **Redisの利用を推奨**します。ルームはTTL（既定: 最終アクセスから2時間）で自動削除されます。

## ローカル開発

```bash
npm install
npm run dev
```

- フロントエンド: http://localhost:5173 （Vite devサーバー。`/socket.io` はNodeへプロキシ）
- サーバー: http://localhost:3000

開発中に動作確認しやすくするには、`.env` でラウンド時間を短縮できます:

```bash
TEXT_ROUND_SECONDS=5
DRAWING_ROUND_SECONDS=15
FORCE_CARD_TYPE=fusion   # normal / effect / fusion を固定（開発時のみ有効）
```

開発モード（`NODE_ENV != production`）ではロビーに**「テスト用CPUを追加」**ボタンが表示され、1人でも全フローを確認できます。これらの開発機能は本番では自動的に無効になります。

## 本番用ビルド

```bash
npm run build   # Reactアプリを dist/ へビルド
npm start       # Express + Socket.io + dist/ を1プロセスで起動
```

`npm start` は `PORT` 環境変数（既定3000）で `0.0.0.0` で待ち受け、ReactアプリとSocket.ioを**同一オリジン**から配信します。`/join/ABCD` などのSPAルートへの直接アクセスにも対応しています。

## 環境変数

`.env.example` をコピーして `.env` を作成してください（秘密情報はリポジトリに含めないこと）。

| 変数 | 既定値 | 説明 |
|---|---|---|
| `PORT` | 3000 | 待ち受けポート（ホスティングから供給される値を使用） |
| `NODE_ENV` | – | `production` で本番モード（開発機能を無効化） |
| `PUBLIC_BASE_URL` | – | 任意。参加URLは基本クライアントが自オリジンから生成 |
| `REDIS_URL` | – | Redis接続先。未設定ならメモリ管理 |
| `ROOM_TTL_SECONDS` | 7200 | ルーム自動破棄までの秒数（最終アクセス起点） |
| `MAX_PLAYERS` | 4 | 最大参加人数（上限4） |
| `TEXT_ROUND_SECONDS` | 90 | テキスト工程（ラウンド1〜7）の制限時間 |
| `DRAWING_ROUND_SECONDS` | 150 | イラスト工程（ラウンド8）の制限時間 |
| `MAX_IMAGE_BYTES` | 1500000 | イラストPNGの最大サイズ |
| `NAME_JOIN_SPACE` | 0 | 1にするとカード名の前半と後半の間に空白を挿入 |
| `FORCE_CARD_TYPE` | – | 【開発専用】カード種別の固定 |

## クラウドへのデプロイ

Node.jsとWebSocketに対応したサービスならどこでも動きます（Render / Railway / Fly.io / Koyeb など）。**1インスタンス構成**を前提としてください（水平スケールには対応していません。Redisは再起動・インスタンス入替時の状態退避用です）。

### 例: Render にデプロイする

1. このリポジトリをGitHubへpushする
2. Render で **New → Web Service** を選び、リポジトリを接続
3. 以下を設定:
   - **Runtime**: Node
   - **Build Command**: `npm install && npm run build`
   - **Start Command**: `npm start`
   - **Node バージョン**: 20以上（`package.json` の `engines` を自動検出、または環境変数 `NODE_VERSION=20`）
   - **PORT**: 設定不要（Renderが自動注入する `PORT` をサーバーが使用します）
4. （推奨）Redisを用意して環境変数を設定:
   - Render の **Key Value（Redis互換）** を作成し、Internal URL を `REDIS_URL` に設定
   - または Upstash などの外部Redisの接続URLを設定
5. デプロイ完了後、発行されたURL（例 `https://your-app.onrender.com`）を開く
6. 確認事項:
   - **HTTPSで開けること**（Renderは自動でTLS終端。同一オリジンなのでWSSも自動）
   - ルーム作成→別端末で参加し、**WebSocket接続**が確立すること（開発者ツールのNetworkで `websocket` を確認。制限がある回線ではSocket.ioが自動的にロングポーリングへフォールバックします）

> **無料プランの注意**: 無料インスタンスは一定時間アクセスがないと**スリープ**し、初回アクセスの起動に数十秒かかります。またスリープ時にメモリ上のルームは消えます（Redis利用で軽減）。常用するなら有料プラン等の常時起動インスタンスを推奨します。

## スマートフォンで遊ぶ方法

1. ホストが公開URL（例 `https://your-app.onrender.com`）をスマホで開く
2. プレイヤー名を入力して「ルームを作る」をタップ
3. ロビーに表示された**QRコード**を他の参加者へ見せる
4. 他の参加者はスマホのカメラでQRコードを読み取る（`/join/XXXX` が開き、コードは自動入力）
5. プレイヤー名を入力して参加する
6. 2〜4人そろったらホストが「ゲームを開始する」をタップ

参加者全員が同じWi-Fiにいる必要はありません。**各スマホがインターネットに接続できれば**（家庭用Wi-Fi・公衆Wi-Fi・モバイル回線いずれでも）参加できます。プレイ中にPCやローカルサーバーを起動しておく必要もありません。

## 同一PCでの開発確認

1. `npm run dev` を実行
2. ブラウザで http://localhost:5173 のタブを2〜4個開く
3. 各タブで異なるプレイヤー名を入力（※同一ブラウザの複数タブはlocalStorageを共有するため、**別々のシークレットウィンドウ／別ブラウザ**を使うと再接続の動作確認が正確になります）
4. 1つのタブでルームを作成し、他のタブからルームコードで参加
5. 全8ラウンドと結果発表まで確認

サーバーロジックの自動テスト（2/3/4人での回転・情報公開・自動提出・再接続・不正提出の拒否・結果発表）:

```bash
npm run smoke
```

## 既知の制約

- 単一Node.jsプロセス前提（複数インスタンスへの水平スケールにはSocket.ioアダプタ等が別途必要）
- Redis利用時もスナップショット保存のため、クラッシュのタイミングによっては直近数秒の下書きが失われる場合がある
- 完成カードのPNG保存機能は未実装（結果一覧のスクリーンショットで代用可能）
